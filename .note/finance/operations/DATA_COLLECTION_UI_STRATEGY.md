# Data Collection UI Strategy

## 목적
이 문서는 현재 `finance` 패키지의 데이터 수집 기능을 사람이 쉽게 실행할 수 있도록,
"버튼 클릭 기반 수집 UI"를 어떤 방식으로 고도화하면 좋은지에 대한 기본 전략을 정리한 문서다.

대상 목표:
- 매일 데이터 수집을 쉽게 실행
- 복잡한 Python 함수 호출 없이 버튼으로 수집 시작
- 어떤 작업이 성공/실패했는지 확인
- 장기적으로는 전략 백테스트 시스템의 운영 콘솔로 확장

---

## 1. 최종 목표 재정의

사용자의 장기 목표는 다음과 같이 이해할 수 있다.

1. 다양한 전략을 입력한다.
2. 전략에 필요한 데이터를 DB에서 읽는다.
3. 필요한 전처리와 분석을 수행한다.
4. 최종적으로 백테스트를 실행한다.

이 목표를 위해서는 먼저 데이터 인프라가 안정적으로 돌아야 한다.

즉, 지금 가장 먼저 필요한 것은:

- 데이터를 매일 쉽게 수집할 수 있는 실행 인터페이스
- 수집 결과를 눈으로 확인할 수 있는 운영 화면

이다.

---

## 2. 지금 만들려는 것의 정체

지금 만들려는 것은 일반 사용자용 서비스라기보다,
**내부 운영용 데이터 수집 콘솔(admin console)** 에 가깝다.

이 관점이 중요하다.

왜냐하면 현재 필요한 것은:
- 예쁜 서비스 화면
- 복잡한 사용자 권한 체계
- 공개 API

가 아니라,

- 버튼 클릭
- 배치 실행
- 로그 확인
- 실패 확인
- 재실행

이기 때문이다.

즉, 1차 목표는 "운영툴"이어야 한다.

---

## 3. 구현 방식 후보

현재 선택지는 크게 세 가지다.

### 옵션 A. 간단한 데스크톱 GUI
예:
- `tkinter`
- `PySide6`

장점:
- 로컬에서 바로 실행 가능
- 버튼 중심 UI 만들기 쉬움
- 웹 서버 지식이 적어도 됨

단점:
- 이후 원격 접속/공유가 불편
- 상태 관리와 확장성이 웹보다 약함
- 로그/테이블/배치 현황 화면을 고도화할수록 불리함

### 옵션 B. 내부용 간단한 웹 앱
예:
- `Streamlit`
- `FastAPI + Jinja/HTML`
- `Flask`

장점:
- 브라우저에서 쉽게 접근 가능
- 버튼, 상태 표시, 테이블 표시가 편함
- 나중에 백테스트 UI까지 확장하기 좋음

단점:
- 웹 실행 구조를 조금 잡아야 함
- 프론트와 백엔드 경계 설계가 필요함

### 옵션 C. CLI + 스케줄러만 먼저
예:
- Python CLI
- cron / launchd / task scheduler

장점:
- 구현이 가장 빠름
- 자동화와 일괄실행에 강함

단점:
- 지금 원하는 "버튼 클릭 운영" 경험과는 거리가 있음
- 상태 확인성이 약함

## 4. 현재 프로젝트에 대한 추천

## 추천 결론
**1차는 내부용 웹 앱으로 시작하는 것이 가장 좋다.**

그 중에서도 현실적인 추천은 다음 두 단계다.

### 1단계 추천
간단한 웹 기반 운영 콘솔

권장 방향:
- UI: `Streamlit` 또는 아주 얇은 `FastAPI` 기반 관리화면
- 실행 로직: 기존 `finance/data/*` 함수 재사용
- 목적: 버튼 클릭으로 적재 작업 실행

### 2단계 추천
운영 콘솔 + 예약 실행 + 실행 이력 관리

권장 방향:
- job runner 분리
- 실행 이력 DB 저장
- 예약 작업 추가

---

## 5. 왜 웹이 GUI보다 더 적합한가

현재 프로젝트 상황을 기준으로 보면 웹이 더 유리하다.

### 이유 1. 이 프로젝트는 결국 “운영 콘솔 + 리서치 UI”로 커질 가능성이 높다
지금은 데이터 수집 버튼만 필요하지만,
다음 단계에서는 자연스럽게 아래가 붙는다.

- 마지막 적재 시각
- 어떤 심볼이 실패했는지
- 어떤 테이블이 몇 행 업데이트됐는지
- 팩터 계산 실행 버튼
- 백테스트 실행 버튼
- 전략 파라미터 입력

이런 것은 데스크톱 GUI보다 웹 UI가 훨씬 잘 맞는다.

### 이유 2. 현재 프로젝트는 Python 중심이라 내부용 웹이 구현 난이도가 높지 않다
특히 내부용 관리 화면은 복잡한 프론트엔드가 없어도 된다.

즉:
- React 같은 큰 프론트엔드 없이도 충분
- Python 함수 호출 결과를 바로 표와 로그로 보여주기 쉬움

### 이유 3. 나중에 자동화와 연결하기 쉽다
버튼 클릭 수집과 스케줄 수집이 결국 같은 job을 호출하도록 만들면,
운영 구조가 깔끔해진다.

---

## 6. 가장 추천하는 1차 구조

가장 추천하는 구조는 아래다.

```text
UI Layer
  - internal web admin
  - buttons / forms / status panels

Job Layer
  - collect universe
  - collect asset profiles
  - collect OHLCV
  - collect fundamentals
  - collect financial statements
  - calculate factors

Service Layer
  - thin wrapper around existing finance/data functions

Persistence Layer
  - MySQL
  - logs/
```

핵심 원칙:
- UI는 직접 DB 적재 로직을 갖지 않는다
- UI는 job 함수를 호출만 한다
- 기존 `finance/data/*` 함수는 최대한 재사용한다

즉, 새로 만들 것은 “수집 로직”보다 “실행 제어 계층”이다.

---

## 7. 추천 기능 범위

1차 버전에서는 아래 정도만 있으면 충분하다.

## 7-1. 대시보드

보여줄 것:
- 최근 실행 작업 목록
- 마지막 성공 시각
- 마지막 실패 시각
- 현재 DB 상태 요약
- 최근 로그 파일 경로 또는 요약

## 7-2. 수집 실행 버튼

버튼 예시:
- `NYSE 종목 목록 수집`
- `자산 프로필 수집`
- `OHLCV 수집`
- `요약 재무제표 수집`
- `상세 재무제표 수집`
- `팩터 계산`
- `전체 수집 파이프라인 실행`

## 7-3. 실행 파라미터 입력

예:
- symbols
- kind (`stock`, `etf`)
- freq (`annual`, `quarterly`)
- start / end
- interval (`1d`, `1wk`, `1mo`)
- chunk size

1차에서는 모든 파라미터를 다 열지 말고,
자주 쓰는 것만 노출하는 것이 좋다.

## 7-4. 실행 결과 표시

보여줄 것:
- 성공 여부
- 처리 심볼 수
- 적재 행 수
- 실패 심볼 수
- 실패 목록 일부
- 소요 시간

---

## 8. 추천 개발 순서

## 1단계
현재 수집 함수들을 "job 함수"로 감싼다.

예:
- `run_collect_asset_profiles(...)`
- `run_collect_ohlcv(...)`
- `run_collect_fundamentals(...)`
- `run_collect_financial_statements(...)`
- `run_calculate_factors(...)`

이 단계의 목적:
- UI가 직접 low-level 함수를 건드리지 않게 함

## 2단계
공통 실행 결과 포맷을 만든다.

예:
- `job_name`
- `started_at`
- `finished_at`
- `status`
- `rows_written`
- `symbols_processed`
- `failed_symbols`
- `message`

이렇게 해야 UI에 결과를 쉽게 뿌릴 수 있다.

## 3단계
내부용 웹 화면을 만든다.

이때 1차는 복잡한 구조가 필요 없다.

필요한 화면:
- dashboard
- collect page
- run result page

## 4단계
실행 이력 저장을 붙인다.

예:
- `ingestion_run_history` 테이블

저장 내용:
- 어떤 job이
- 언제 실행됐고
- 성공했는지
- 몇 건 처리했는지
- 실패 심볼이 있었는지

## 5단계
예약 실행을 붙인다.

예:
- 하루 1회 OHLCV
- 주 1회 asset profile
- 분기/월 단위 재무제표 업데이트

---

## 9. 웹 프레임워크 추천

## 현실적 추천 1: Streamlit

가장 빨리 만들고 검증하기 좋다.

좋은 점:
- Python만으로 UI 작성 가능
- 버튼, 폼, 테이블, 로그 출력이 빠름
- 내부 운영툴에 매우 적합

아쉬운 점:
- 서비스 구조가 커지면 정교한 앱 구조로 바꾸고 싶어질 수 있음

판단:
- **프로토타입과 1차 운영툴로 매우 적합**

## 현실적 추천 2: FastAPI + 간단한 HTML

좋은 점:
- 구조가 더 오래 간다
- API와 UI를 분리하기 쉽다
- 나중에 백테스트 실행 API까지 확장하기 좋다

아쉬운 점:
- 1차 구현 속도는 Streamlit보다 느릴 수 있다

판단:
- **장기 구조로는 더 좋지만, 1차 진입 장벽은 약간 더 높다**

## 비추천 1차안: 무거운 프론트엔드

예:
- React 단독 프론트 + 별도 API 서버

이유:
- 지금 단계에서는 과하다
- 핵심 문제는 UI 화려함이 아니라 운영 흐름 정리다

## 10. 추천 운영 시나리오

가장 현실적인 1차 시나리오는 아래다.

### 매일 아침
사용자가 웹 페이지 접속

### 화면에서 확인
- 어제 마지막 수집 성공 여부
- 최근 실패 작업
- 오늘 아직 실행 안 된 작업

### 버튼 클릭
- `OHLCV 수집 실행`
- 필요 시 `재무제표 수집 실행`
- 필요 시 `팩터 계산 실행`

### 실행 후 확인
- 몇 건 적재됐는지
- 실패 심볼이 무엇인지
- 로그 링크 또는 로그 요약

이 흐름이 먼저 안정화되면 그 다음에 자동 스케줄러를 붙이는 것이 좋다.

---

## 11. 지금 단계에서 같이 설계해야 할 백엔드 구조

UI를 만들기 전에 아래는 먼저 정리하는 것이 좋다.

### A. job 함수 표준화
현재 함수들은 개별 적재 함수 단위다.

추가되면 좋은 것:
- UI가 호출할 표준 wrapper
- 반환 형식 통일

### B. 실행 이력 구조
최소한 아래는 남겨야 한다.

- 작업명
- 시작/종료 시각
- 상태
- 처리 건수
- 실패 건수
- 에러 메시지 요약

### C. 로그 접근 방식
지금은 `logs/` 파일이 중심이다.

1차는 이걸 그대로 써도 된다.

다만 UI에서는:
- 최신 로그 파일명
- 최근 20줄 요약

정도는 보여주는 것이 좋다.

### D. 파이프라인 순서 정의
전체 수집 버튼을 누르면 어떤 순서로 실행할지 정해져야 한다.

추천 순서:
1. NYSE universe
2. asset profile
3. OHLCV
4. fundamentals
5. financial statements
6. factors

## 12. 지금 당장 추천하는 구현 전략

가장 추천하는 전략은 아래다.

### 전략 A. 1차는 내부 웹 운영 콘솔

목표:
- 버튼 클릭으로 수집 실행
- 결과 확인
- 실패 확인

구현:
- `app/` 또는 `admin/` 디렉터리 추가
- 웹 UI 추가
- job wrapper 추가

### 전략 B. 기존 finance/data 함수는 최대한 그대로 사용

즉:
- 새로 수집 로직을 다시 쓰지 않음
- UI는 기존 함수 실행을 감쌈

### 전략 C. 자동화는 2차로 미룸

지금은:
- 수동 버튼 실행
- 결과 확인

이 먼저다.

자동 스케줄은 운영 흐름이 안정화된 뒤 붙이는 것이 좋다.

## 13. 내가 추천하는 실제 우선순위

지금부터 실제 구현 논의를 시작한다면 아래 순서를 권장한다.

### 1순위
내부용 웹 콘솔 방식을 확정한다.

추천:
- Streamlit 1차안

### 2순위
job wrapper와 공통 실행 결과 포맷을 설계한다.

### 3순위
OHLCV / fundamentals / factors 세 가지부터 버튼화한다.

이유:
- 현재 가장 실용적
- 기존 코드와 연결이 분명함

### 4순위
상세 재무제표와 asset profile, universe sync를 추가한다.

### 5순위
실행 이력 저장과 스케줄러를 붙인다.

## 14. 한 줄 결론

지금 단계에서 가장 좋은 방향은:

**기존 `finance/data/*` 적재 함수를 재사용하는 내부용 웹 관리 콘솔을 먼저 만들고, 버튼 실행 -> 실행 결과 표시 -> 실행 이력 저장 순서로 확장하는 것**이다.

이 방식이 가장 빠르고, 현재 프로젝트의 다음 단계인 "DB 기반 전략 실행 플랫폼"으로도 자연스럽게 이어진다.
