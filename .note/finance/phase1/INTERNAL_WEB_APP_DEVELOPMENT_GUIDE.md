# Internal Web App Development Guide

## 목적
이 문서는 `finance` 패키지의 데이터 수집 기능을 쉽게 실행하기 위한 내부용 웹 앱을 만들 때,
실제로 바로 따라갈 수 있는 개발 순서를 단계별로 정리한 가이드다.

목표는 다음과 같다.

- 버튼 클릭으로 데이터 수집 실행
- 실행 결과와 실패 상황 확인
- 장기적으로 백테스트 실행 콘솔로 확장 가능한 구조 확보

이 문서는 1차 구현 기준으로 작성한다.

---

## 최종 방향

1차 구현 방향:
- 내부용 웹 앱
- 기존 `finance/data/*` 수집 함수 재사용
- 새로 만들 것은 UI와 job wrapper 계층

권장 기술 방향:
- 1차: `Streamlit`
- 이후 확장 시: 필요하면 `FastAPI` 계열로 재구성 가능

---

## 전체 개발 순서

1. 범위 확정
2. job wrapper 설계
3. 공통 실행 결과 포맷 정의
4. 웹 앱 기본 골격 생성
5. 개별 수집 버튼 연결
6. 실행 결과 표시
7. 로그/실패 확인 화면 추가
8. 실행 이력 저장
9. 전체 파이프라인 실행 버튼 추가
10. 운영 안정화 후 스케줄러 검토

---

## 1단계. 1차 범위 확정

### 목표
처음부터 모든 기능을 넣지 않고, 1차 앱이 무엇을 해야 하는지 확정한다.

### 1차 범위 권장안
- OHLCV 수집 실행
- 요약 재무제표 수집 실행
- 팩터 계산 실행
- 최근 실행 결과 확인
- 실패 심볼/에러 요약 확인

### 1차에서 제외 권장
- 전략 백테스트 UI
- 상세 재무제표 복잡한 조회 화면
- 사용자 인증
- 멀티유저 기능
- 자동 스케줄러

### 산출물
- 1차 기능 목록 확정
- 제외 범위 목록 확정

### 검증 기준
- “지금 바로 써야 하는 운영 기능”만 남아 있는가
- 구현 범위가 1~2회 작업 사이클 안에 끝날 정도로 작아졌는가

---

## 2단계. Job Wrapper 계층 설계

### 목표
웹 UI가 `finance/data/*`의 low-level 함수를 직접 호출하지 않도록 중간 실행 계층을 만든다.

### 해야 할 일
- `finance` 하위 또는 별도 `app/` 하위에 job wrapper 모듈 추가
- 각 수집 함수에 대응하는 wrapper 정의

예시:
- `run_collect_ohlcv(...)`
- `run_collect_fundamentals(...)`
- `run_collect_factors(...)`
- `run_collect_asset_profiles(...)`
- `run_collect_financial_statements(...)`

### wrapper의 역할
- 입력 파라미터 정리
- 기존 함수 호출
- 예외 처리
- 공통 결과 포맷으로 반환

### 산출물
- job 실행 모듈 1개 이상
- 각 job 함수 초안

### 검증 기준
- UI 없이 Python 코드만으로 각 job을 독립 실행할 수 있는가
- wrapper가 기존 `finance/data/*` 로직을 재사용하는가

---

## 3단계. 공통 실행 결과 포맷 정의

### 목표
모든 수집 작업이 동일한 형식의 실행 결과를 반환하도록 만든다.

### 권장 결과 필드
- `job_name`
- `status`
- `started_at`
- `finished_at`
- `duration_sec`
- `symbols_requested`
- `symbols_processed`
- `rows_written`
- `failed_symbols`
- `message`

### 권장 상태값
- `success`
- `partial_success`
- `failed`

### 산출물
- 공통 result dict 또는 dataclass

### 검증 기준
- 서로 다른 job 결과를 동일한 UI 컴포넌트에서 표시할 수 있는가

---

## 4단계. 웹 앱 기본 골격 생성

### 목표
실행 가능한 최소 웹 앱을 만든다.

### 권장 구조 예시
```text
app/
  web/
    main.py
    pages/
      dashboard.py
      collect.py
  jobs/
    ingestion_jobs.py
```

또는 단순하게 시작하면:
```text
admin/
  streamlit_app.py
  jobs.py
```

### 화면 최소 구성
- 대시보드
- 수집 실행 화면

### 산출물
- 웹 앱 실행 엔트리포인트
- 기본 레이아웃

### 검증 기준
- 로컬에서 웹 앱이 정상 실행되는가
- 최소 1개 버튼과 1개 상태 영역이 보이는가

---

## 5단계. 개별 수집 버튼 연결

### 목표
가장 중요한 데이터 작업들을 버튼 클릭으로 실행 가능하게 만든다.

### 우선 연결 순서 권장
1. OHLCV 수집
2. fundamentals 수집
3. factors 계산

그 다음:
4. asset profile 수집
5. detailed financial statements 수집

### 이유
- 현재 프로젝트의 실질적 핵심 파이프라인과 가장 직접 연결됨
- 기존 코드가 상대적으로 정리돼 있음

### 산출물
- 버튼별 실행 핸들러
- 파라미터 입력 폼

### 검증 기준
- 버튼 클릭 시 실제 적재 함수가 실행되는가
- 성공/실패 결과가 UI에 표시되는가

---

## 6단계. 실행 결과 표시

### 목표
사용자가 작업 결과를 바로 이해할 수 있게 한다.

### 표시해야 할 것
- 작업명
- 시작/종료 시각
- 상태
- 처리 건수
- 적재 건수
- 실패 건수
- 요약 메시지

### UI 방식 예시
- 상단 알림 박스
- 결과 카드
- 하단 상세 테이블

### 산출물
- 공통 결과 렌더링 컴포넌트

### 검증 기준
- 작업 성공/실패를 사용자가 즉시 구분할 수 있는가
- 실패 시 원인 추적에 필요한 최소 정보가 보이는가

---

## 7단계. 로그 및 실패 확인 화면 추가

### 목표
실패 작업을 다시 파악하고 재실행할 수 있게 한다.

### 표시 대상
- 최근 로그 파일 목록
- 최근 실패 심볼
- 최근 에러 메시지 일부

### 1차 구현 권장
- `logs/` 파일 직접 읽기
- 최근 파일 1~3개 요약 표시

### 산출물
- 로그/실패 요약 영역

### 검증 기준
- 실패 시 UI만 보고 다음 조치가 가능한가
- 최소한 어떤 job에서 어떤 심볼이 실패했는지 보이는가

---

## 8단계. 실행 이력 저장

### 목표
앱을 닫아도 이전 실행 결과를 다시 볼 수 있게 한다.

### 권장 방식
- 1차: JSON 또는 Markdown 로그도 가능
- 더 좋은 방식: MySQL에 실행 이력 테이블 추가

### 이력 필드 예시
- `job_name`
- `status`
- `started_at`
- `finished_at`
- `duration_sec`
- `rows_written`
- `failed_count`
- `message`

### 산출물
- 실행 이력 저장 방식 확정
- 이력 조회 기능

### 검증 기준
- 마지막 실행 결과를 앱 재시작 후에도 볼 수 있는가

---

## 9단계. 전체 파이프라인 실행 버튼 추가

### 목표
개별 작업뿐 아니라 자주 쓰는 전체 수집 시퀀스를 한 번에 실행한다.

### 권장 파이프라인 순서
1. NYSE universe
2. asset profile
3. OHLCV
4. fundamentals
5. financial statements
6. factors

### 주의
- 1차 전체 실행은 너무 길 수 있으므로,
  먼저 아래 단축 시퀀스를 추천한다.

추천 1차 시퀀스:
1. OHLCV
2. fundamentals
3. factors

### 산출물
- pipeline job wrapper
- 전체 실행 버튼

### 검증 기준
- 순차 실행이 실패 시 어디서 멈췄는지 보이는가
- partial success를 구분할 수 있는가

---

## 10단계. 운영 안정화

### 목표
수동 버튼 실행만 가능한 상태에서 운영 가능한 상태로 끌어올린다.

### 추가할 것
- 기본 파라미터 preset
- 최근 사용 파라미터 기억
- 실패한 작업 재실행
- 너무 긴 작업에 대한 진행 상태 표시

### 산출물
- 사용성 보강

### 검증 기준
- 사용자가 코드를 열지 않고 반복 실행할 수 있는가

---

## 11단계. 자동화 검토

### 목표
수동 운영이 안정화된 뒤 자동 실행을 붙인다.

### 권장 시점
- 수동 실행 흐름이 충분히 안정된 뒤
- 실패 원인과 재실행 방식이 정리된 뒤

### 자동화 예시
- 매일 OHLCV
- 주기적 fundamentals
- 필요 시 factors 재계산

### 검증 기준
- 자동 실행이 실패해도 복구 경로가 명확한가

---

## 추천 1차 개발 순서 요약

가장 현실적인 구현 순서는 아래다.

4. Streamlit 앱 골격 생성
5. OHLCV 버튼 연결
6. fundamentals 버튼 연결
7. factors 버튼 연결
8. 결과/실패 표시

10. 전체 실행 버튼

## 각 단계별 완료 기준

### Phase 1 완료 기준
- 웹 앱이 뜬다
- 버튼 3개가 있다
- OHLCV/fundamentals/factors가 실행된다
- 결과가 화면에 나온다

### Phase 2 완료 기준
- 실패 요약을 볼 수 있다
- 실행 이력이 남는다
- 전체 실행 버튼이 동작한다

### Phase 3 완료 기준
- asset profile, detailed financial statements까지 UI에 연결된다
- 반복 운영이 가능하다

---

## 지금 바로 시작할 때 권장하는 첫 작업

지금 실제 구현을 시작한다면 아래 순서가 가장 좋다.

1. `Streamlit` 사용 여부 확정
2. 앱 디렉터리 구조 확정
3. `run_collect_ohlcv`, `run_collect_fundamentals`, `run_collect_factors` wrapper 작성
4. 버튼 3개만 있는 최소 앱 구현

즉, 처음 목표는 “완성된 운영 콘솔”이 아니라:

**OHLCV / fundamentals / factors를 버튼으로 돌릴 수 있는 최소 웹 앱**

이다.

---

## 한 줄 결론

가장 좋은 시작점은:

**기존 수집 함수를 감싸는 job wrapper를 먼저 만들고, 그 위에 Streamlit 기반의 최소 내부 운영 웹 앱을 올리는 것**이다.
