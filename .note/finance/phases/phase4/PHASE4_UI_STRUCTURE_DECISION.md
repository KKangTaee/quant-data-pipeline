# Phase 4 UI Structure Decision

## 목적
이 문서는 Phase 4 첫 구현에서
백테스트 UI를 어디에 어떻게 붙일지에 대한 구조 결정을 기록한다.

---

## 결정 결과

선택 결과:
- **메인 앱 하나 + 탭별/속성별 모듈 분리**

대상 경로:
- 메인 엔트리: `app/web/streamlit_app.py`
- 내부 구조: `app/web/*` 또는 `app/web/pages/*` 형태의 탭별 모듈 분리

---

## 검토한 선택지

### 1. 기존 `streamlit_app.py`에 통합

장점:
- 가장 빠르게 붙일 수 있다
- 기존 Streamlit 진입점을 그대로 재사용한다

단점:
- 수집 UI와 백테스트 UI가 한 파일에서 빠르게 복잡해진다
- ingestion 운영 로직과 전략 실행 로직이 섞이기 쉽다
- 이후 유지보수 시 역할 경계가 흐려질 수 있다

### 2. 별도 `backtest_app.py`로 분리

장점:
- 수집 앱과 백테스트 앱의 책임이 분명히 분리된다
- Phase 4에서 실험적인 UI를 비교적 안전하게 붙일 수 있다
- 이후 runtime wrapper와 result bundle을 백테스트 앱 관점으로 정리하기 쉽다

단점:
- 앱 진입점이 2개가 된다
- 공통 컴포넌트/헬퍼가 필요해질 수 있다

### 3. Streamlit multipage 구조로 재편

장점:
- 장기적으로는 가장 정돈된 사용자 경험을 줄 수 있다

단점:
- 지금 시점에는 리팩터링 범위가 커진다
- 기존 ingestion UI까지 같이 손봐야 할 가능성이 높다

---

## 왜 한 앱 + 탭 구조가 적절한가

현재 프로젝트 상태를 기준으로 보면:

- 수집 운영 UI는 이미 별도의 목적과 흐름이 있다
- Phase 4는 전략 실행 / 결과 확인이라는 다른 제품 흐름을 여는 단계다
- 사용자는 한 앱 안에서 수집과 분석을 자연스럽게 오가길 원한다

따라서 제품 구조는 한 앱으로 유지하되,
코드 구조는 탭별/속성별로 나누는 것이 가장 현실적이다.

즉 구조적으로는:

```text
app/web/streamlit_app.py          -> 메인 엔트리 / 탭 라우팅
app/web/pages/ingestion.py        -> 수집 탭
app/web/pages/backtest.py         -> 백테스트 탭
app/web/pages/...                 -> 이후 확장 탭
```

처럼 가는 것이 현재 단계에 맞다.

---

## 구현 영향

이 결정에 따라:

1. 기존 `streamlit_app.py`는 메인 엔트리로 유지
2. 수집과 백테스트는 탭으로 분리
3. 내부 구현은 탭별/속성별 모듈로 분리
4. runtime wrapper / result bundle은 백테스트 탭 기준으로 붙인다
5. 공통 로직이 생기면 이후 `app/web/common/*` 또는 유사한 위치로 분리 검토

---

## 현재 판단

이 결정은 first pass에 적합하다.

장기적으로 multipage 구조가 더 적합해질 수는 있지만,
지금은 역할 분리와 구현 속도 사이의 균형이 더 중요하다.

즉 현재 Phase 4의 기준 구조는:
- **앱은 하나**
- **기능은 탭으로 분리**
- **코드는 탭별/속성별 모듈로 분리**

로 본다.
