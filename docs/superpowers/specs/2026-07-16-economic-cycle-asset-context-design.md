# 경제 사이클 자산별 확인 포인트 설계

## 목적

현재의 `시장의 다음 질문`은 네 자산군에 같은 국면명과 정적인 설명만 반복해 사용자가 실제로 무엇을 확인해야 하는지 알기 어렵다.
이를 경제 사이클 근거를 자산별로 다시 읽는 `자산별 확인 포인트`로 바꾼다.

## 사용자 흐름

1. 사용자는 자산군별 `우호 / 부담 / 혼재 / 자료 부족` 상태를 먼저 읽는다.
2. 각 카드에서 상태를 만든 상위 근거 두 개를 확인한다.
3. `바뀌는 조건`에서 다음 데이터 갱신 때 무엇을 재확인할지 이해한다.

## 판정 계약

- 입력은 현재 economic-cycle evidence의 네 canonical factor다.
  - `activity_score`
  - `labor_income_score`
  - `financial_leading_score`
  - `inflation_policy_score`
- factor 값은 기존 `강화 / 약화 / 중립` 기준을 그대로 사용한다.
- 자산별 orientation table로 각 factor를 `우호 / 부담 / 중립`로 번역한다.
- 인식 가능한 비중립 factor가 두 개 미만이면 `자료 부족`이다.
- 우호와 부담의 개수 차이가 두 개 이상일 때만 `우호` 또는 `부담`으로 판정한다.
- 그 외에는 `혼재`다.
- 국면명은 설명 맥락으로만 사용하며 factor 점수에 중복 반영하지 않는다.
- 결과는 수익률 전망, 목표가격, 매수·매도 지시가 아니다.

## 자산별 해석

| 자산군 | 실물·고용 강화 | 금융·선행 강화 | 물가·정책 압력 강화 |
|---|---|---|---|
| 채권·금리 | 부담 | 부담 | 부담 |
| 주식 | 우호 | 우호 | 부담 |
| 금·달러 | 부담 | 부담 | 우호 |
| 원자재 | 우호 | 우호 | 우호 |

약화는 표의 반대 방향으로 해석한다.
`금·달러`는 두 자산의 가격 방향을 합친 예측이 아니라 위험회피·실질금리·정책 불확실성을 확인하는 방어 맥락이다.

## UI

- 제목: `자산별 확인 포인트`
- 보조 문구: `매매 신호가 아닌 조건부 점검`
- desktop/tablet: 2×2 카드
- mobile: 1열 카드
- 카드 구성:
  - 자산군명과 상태 배지
  - `향후 우세 국면 · 상태` 요약
  - 한 문장 해석
  - 상위 근거 두 개와 각 근거의 우호/부담 표시
  - `바뀌는 조건`

## 완료 조건

- 네 자산군이 실제 evidence에 따라 서로 다른 상태를 가질 수 있다.
- evidence 부족 시 이유를 만들지 않고 `자료 부족`을 표시한다.
- 기존 API 소비자를 위해 `context`와 `is_directional_forecast=false`를 유지한다.
- Python 테스트, React build, desktop/mobile Browser QA를 통과한다.
