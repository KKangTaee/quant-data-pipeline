# Validation Evidence Audit

Last updated: 2026-06-01

## Scope

This is a read-only audit of the current `main-dev` worktree for the flow:

`Backtest Analysis -> Practical Validation -> Final Review -> Selected Portfolio Dashboard`

The audit reviews whether validation evidence is investment-grade, proxy/prototype, or policy/heuristic. It does not approve live investment, broker/account integration, order creation, or automatic rebalancing.

No code, JSONL registry, saved setup, candidate record, or run-history artifact was modified for this audit. The only intended write is this research note and companion source notes under `.aiworkspace/note/finance/researches/active/2026-06-validation-evidence-audit/`.

## Executive Conclusion

The product has a stronger evidence chain than a simple backtest screen. It explicitly distinguishes result integrity, runtime replay, benchmark parity, provider/data coverage, construction risk, cost realism, selected-route gate, open review items, and selected-dashboard follow-up. The strongest parts can be trusted as workflow gates: source contract checks, Data Trust blockers, period coverage, benchmark parity, actual runtime replay for supported strategies, required-module `NOT_RUN`/`NEEDS_INPUT` blocking, selection-route blockers, and selected-dashboard open issue/deployment preflight surfacing.

The product should not yet present the full chain as "real-money validated" or "deployment ready". Several items are still policy heuristics or post-hoc robustness evidence: promotion policy, Real-Money/Execution Preview labels, liquidity capacity from current provider snapshots, fixed-bps transaction cost model, cost/slippage sensitivity if not explicitly computed, walk-forward/OOS/regime checks when run after candidate selection, risk contribution proxy, component role semantics, and dashboard deployment readiness. These should be labeled as "evidence", "review required", or "prototype/proxy" unless the specific run has actual replay, actual provider coverage, historical lifecycle/survivorship evidence, net-cost proof, and explicit cost/slippage sensitivity.

The most important strict-risk finding: the selected-route gate is stricter than the Stage 1 and module handoff gates, but it still allows `WATCH` and open review items to proceed into the Selected Dashboard. That is acceptable for a monitoring candidate workflow if the UI keeps "Selected Dashboard" separate from live approval. It is not sufficient as an investment execution gate.

Local DB spot checks show that the price, ETF provider, holdings / exposure, and FRED macro tables contain usable recent evidence for covered symbols. They do not prove universal data correctness. The strongest current DB evidence is price-window coverage and current provider freshness. The weakest DB evidence for investment-grade selection is broad historical universe / survivorship control: the local lifecycle table is mostly current-listing partial evidence, with only a tiny actual historical-listing slice in this worktree.

## Status Vocabulary

The code uses module-specific routes, but the practical meaning is consistent enough for audit:

| Status | Practical meaning for this audit |
| --- | --- |
| `PASS` | Evidence exists and the coded policy considers the item usable. It is not automatically proof of future performance or live tradability. |
| `REVIEW` | Evidence exists but is incomplete, weak, stale, proxy, policy-threshold-based, or requires operator judgment. This is not a pass. |
| `NEEDS_INPUT` | Required evidence is missing, not run, unavailable, or insufficient to evaluate. This should block strict selection when it touches core evidence. |
| `BLOCKED` | Contract, data, source, or policy condition prevents relying on the candidate until fixed. |
| `NOT_RUN` | Used mostly by diagnostics/module plans. It is not a pass; required `NOT_RUN` blocks Practical Validation movement. |

## Currently Trustworthy Checks

These can be trusted as real workflow or technical validation gates, within their stated boundaries:

| Check | Why it is currently trustworthy | Boundary |
| --- | --- | --- |
| Data Trust / source contract blockers | Reads concrete result metadata, price freshness, warnings, component and weight contract state. | Does not prove PIT, survivorship, or provider truth. |
| Runtime replay for supported strategies | Re-runs supported strategy contracts through the runtime and combines weighted mixes from replayed outputs. | Current DB can differ from original data; unsupported strategies still need separate proof. |
| Runtime period coverage | Compares requested and actual validation periods. | 7-day tolerance is policy, not market-risk proof. |
| Benchmark parity | Checks same period, frequency, and coverage for portfolio vs benchmark curves. | Does not validate benchmark appropriateness. |
| Required module gate | Blocks required `BLOCKED`/`NEEDS_INPUT`/`NOT_RUN` modules before Final Review movement. | Allows `REVIEW` movement; Final Review must keep review visible. |
| Selected-route gate | Blocks selected save on selection blockers and review-required issues. | Allows WATCH/open items and is not live approval. |
| Net cost curve proof when measurable | Requires gross/net and estimated cost metadata, not only a cost assumption. | Still only proves the configured fixed-cost model. |
| Data coverage audit when lifecycle rows are actual/covering | Explicitly distinguishes current snapshot vs historical/lifecycle evidence. | PASS is only as strong as loader/source provenance. |
| Open review item handoff | Carries WATCH/deployment review items to the Selected Dashboard. | It is a warning surface, not a correction. |

## Prototype / Proxy / Heuristic Labels Required

These should be visibly labeled as prototype/proxy/heuristic unless a specific run provides stronger evidence:

| Area | Label to use | Reason |
| --- | --- | --- |
| Real-Money / promotion policy | `policy signal`, `shortlist signal`, or `execution preview` | It combines thresholds and current snapshots; it is not live readiness. |
| Liquidity / operability | `current provider snapshot / capacity proxy` | AUM, ADV, spread, and freshness do not model trade size, market impact, or fill quality. |
| Walk-forward / OOS / regime | `post-hoc robustness evidence` | They are real calculations, but not independent OOS unless the candidate selection process sealed the holdout. |
| Risk contribution | `volatility contribution proxy` | Uses component monthly returns and proxy/mixed source handling; no full marginal contribution/covariance artifact. |
| Component role / weight | `metadata discipline check` | Role fit and weight rationale are semantic source metadata, not quantitative validation. |
| Cost/slippage sensitivity | `prototype unless explicit axis computed` | Generic robustness is not the same as cost/slippage sensitivity. |
| Provider coverage freshness | `validation-time snapshot` | Provider snapshots are current evidence, not guaranteed historical point-in-time holdings/exposure. |
| Dashboard deployment readiness | `read-only deployment evidence preflight` | It has no account, broker, live approval, order, or automatic rebalance authority. |

Current snapshots are valid current evidence, not historical PIT evidence. Broad equity survivorship control is not yet strong enough in this worktree to be treated as a pass without actual lifecycle rows covering the candidate period.

## Handoff Integrity To Selected Dashboard

The handoff does not directly rewrite validation failures into passes. It preserves:

- `open_review_items` from Final Review into Selected Dashboard follow-up.
- deployment-readiness policy snapshot rows.
- dashboard `operation_status` states such as `watch`, `re_review_needed`, and `blocked`.
- explicit execution boundaries: no live approval, no order instruction, no broker sync, no auto rebalance.

The remaining risk is product semantics. A row appearing in "Selected Portfolio Dashboard" can imply a stronger endorsement than the underlying evidence supports. The dashboard should keep a first-viewport distinction between:

- "selected for monitoring / practical portfolio tracking"
- "ready for live approval" - currently out of scope and unsupported
- "ready for order / auto rebalance" - currently out of scope and unsupported
